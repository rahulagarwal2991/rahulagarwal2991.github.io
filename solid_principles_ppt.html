üìùThe Rule: One Class, One Job!

Every class should have only one reason to change. Think of it like this:

‚ùå Bad Example - A Swiss Army Knife Class

‚úÖ Good Example - Focused Classes
class Student { // Only handles student data } class StudentDatabase { // Only handles saving/loading } class EmailService { // Only handles emails } class ReportGenerator { // Only handles reports }

üîíOpen for Extension, Closed for Modification

L - Liskov Substitution Principle

üîÑSubtypes Must Be Substitutable

If you have a parent class, you should be able to replace it with any of its child classes without breaking your program.

‚ùå Bad Example - Breaking the Contract
class Bird { fly() { // All birds can fly, right? } } class Penguin extends Bird { fly() { throw new Error("Penguins can't fly!"); // üò± Breaks the contract! } } // This will crash! function makeBirdFly(bird) { bird.fly(); // Works for Bird, but crashes for Penguin }
‚úÖ Good Example - Proper Substitution
class Bird { move() { // General movement } } class FlyingBird extends Bird { move() { this.fly(); } fly() { // Flying birds can fly } } class Penguin extends Bird { move() { this.swim(); } swim() { // Penguins can swim } } // This works for all birds! function makeBirdMove(bird) { bird.move(); // Works for any type of bird }

I - Interface Segregation Principle

üéØMany Small Interfaces > One Big Interface

Don't force classes to implement methods they don't need. Keep interfaces focused and specific.

‚ùå Bad Example - Fat Interface

‚úÖ Good Example - Focused Interfaces
interface Worker { work(); } interface Eater { eat(); } interface Sleeper { sleep(); } interface Flyer { fly(); } interface Swimmer { swim(); } class Human implements Worker, Eater, Sleeper { work() { // ‚úÖ Only implement what humans can do } eat() { // ‚úÖ } sleep() { // ‚úÖ } } class Superman implements Worker, Eater, Sleeper, Flyer { // Superman can do everything humans can do, plus fly! }

D - Dependency Inversion Principle

üîåDepend on Abstractions, Not Concretions

High-level modules shouldn't depend on low-level modules. Both should depend on abstractions.

‚ùå Bad Example - Tight Coupling
class EmailService { sendEmail(message) { // Sends email } } class NotificationManager { constructor() { this.emailService = new EmailService(); // üò± Tightly coupled! } sendNotification(message) { this.emailService.sendEmail(message); // What if we want to send SMS or push notifications? // We'd have to modify this class! üò± } }
‚úÖ Good Example - Loose Coupling
interface NotificationService { send(message); } class EmailService implements NotificationService { send(message) { // Send email } } class SMSService implements NotificationService { send(message) { // Send SMS } } class NotificationManager { constructor(notificationService) { this.notificationService = notificationService; // ‚úÖ Flexible! } sendNotification(message) { this.notificationService.send(message); // Works with ANY notification service! üéâ } }
